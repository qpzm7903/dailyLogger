## Challenges and Growth Areas

### 1. Composable Integration Verification

**Problem:** PERF-003 created `useVirtualScroll.ts` composable but `ScreenshotGallery.vue` wasn't using it.

**Root Cause:** Creating a composable ≠ using it. No verification step to confirm integration.

**Fix Applied:** Review caught this; composable was integrated during review.

**Lesson:** Dev Checklist should include "Verify composable is imported and used in target component."

### 2. Implicit Command Registration

**Problem:** PERF-004 defined `#[command]` in `records.rs` but forgot to register in `main.rs`. Frontend couldn't call the new API.

**Root Cause:** `#[command]` macro creates an impression of auto-registration, but Tauri requires explicit `.register()` call.

**Fix Applied:** Added registration during review.

**Lesson:** Dev Checklist should include "Verify all new commands registered in main.rs."

### 3. CSS Variable Migration Incomplete

**Problem:** PERF-006 established CSS variable theme infrastructure, but 371 hardcoded color references remain across 46 files.

**Impact:** Light theme will show dark elements in components using hardcoded colors like `bg-darker`, `text-white`, `border-gray-700`.

**Current State:** Core theme switching works, but component migration is a separate refactoring effort.

**Lesson:** Theme infrastructure and component migration are two distinct tasks; infrastructure alone doesn't achieve AC4.

### 4. Hardcoded Strings in Multiple Layers

**Problem:** Multiple stories had hardcoded UI strings:
- PERF-002: OnboardingModal hardcoded Chinese
- PERF-005: App.vue 6 hardcoded Chinese in showSuccess/showError calls

**Root Cause:** Each story focused on its core acceptance criteria; i18n was treated as "nice to have."

**Lesson:** If i18n is part of epic scope, i18n verification should be explicit in every story's AC.

## Key Insights

### Technical Insights

1. **CSS Variable + Class Switching is Lightweight Theming**
   - Compared to CSS-in-JS or SCSS variables, Tailwind 4 `@theme` mechanism is simpler
   - Theme switch is just adding/removing a class on root element
   - But: infrastructure ≠ achieved acceptance criteria; component migration is required

2. **Command Registration is an Implicit Dependency**
   - `#[command]` creates the command but doesn't auto-register it
   - This pattern is easy to forget and hard to test (code compiles fine, runtime fails)
   - Solution: Add to Dev Checklist

3. **Composable Pattern Requires Integration Verification**
   - Creating `useX.ts` composable doesn't mean it's used
   - Need explicit check: "Is this composable imported and used in the target component?"

4. **localStorage Persistence Pattern is Reusable**
   - `dailylogger-locale` → `dailylogger-theme`
   - `setLocale()` structure → `setTheme()` structure
   - Pattern consistency reduces cognitive load

5. **Code Review Catches Architecture Issues**
   - Virtual scroll not integrated, command not registered, duplicate CSS definitions
   - These are the kinds of issues that pass local testing and only surface in production
   - Review by different LLM with fresh context is essential

### Process Insights

1. **Dev Checklist Prevents Omissions**
   - Command registration check
   - Duplicate CSS selector check
   - Composable integration check
   - These should be added to story development workflow

2. **Cross-Story Dependencies Need Explicit Tracking**
   - PERF-002 (OnboardingModal) hardcoded strings affect PERF-005 (i18n)
   - PERF-003 (virtual scroll) component affects future theme migration
   - When one story creates debt for another, it should be explicitly noted

3. **AC4 Component Coverage Requires Component Migration**
   - Story providing infrastructure (CSS variables) ≠ AC satisfied
   - If AC requires component coverage, component migration must be a subtask
