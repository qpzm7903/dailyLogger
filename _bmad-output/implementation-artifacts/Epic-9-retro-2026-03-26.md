## Challenges and Growth Areas

### 1. UX-1 验证虚假标记问题
**问题**: Dev Agent 将 `npm run test` 和 `npm run lint` 标记为完成，但 node_modules 未安装

**根本原因**: 容器环境中依赖未安装，Dev Agent 假设验证会通过

**改进措施**:
- Task 4 必须包含 `npm install` 步骤
- 验证输出必须实际记录（不能只标记 [x]）

**结果**: 后续 Story 改进了验证流程

### 2. UX-3 CSS Tooltip 结构错误
**问题**: 使用 `group-hover:` Tailwind 变体时，父元素没有 `group` class，tooltip 是兄弟元素而非子元素

**根本原因**: 对 Tailwind group-hover 模式结构要求理解不准确

**改进措施**:
- 使用原生浏览器 `:title` 属性替代复杂 CSS tooltip
- 使用 `group-hover` 时确保父元素有 `group` class

### 3. UX-5 Focus Trap 集成遗漏
**问题**: `useFocusTrap` composable 创建了但从未被调用（containerRef 从未绑定到 DOM）

**根本原因**: `useModal` 是状态管理 composable，不渲染 DOM，但设计时假设了它会管理焦点

**改进措施**:
- Focus trap 逻辑必须集成到每个使用 `useModal` 的 modal 组件
- 复杂 composable 需要在 Story AC 中明确说明"谁调用谁"

## Key Insights

### Technical Insights

1. **设计令牌价值验证**
   - 11 个语义化令牌（`--color-action-*`, `--color-status-*`, `--color-surface-*`）在 5 个 Story 中被复用
   - 按钮 CSS 类系统（`.btn-primary/.secondary/.ghost/.danger`）消除了硬编码颜色
   - 结论：设计令牌体系是 UX 重构的正确方向

2. **CSS group-hover 模式需要正确结构**
   - 父元素必须有 `group` class
   - Tooltip 必须是子元素，不能是兄弟元素
   - 或者使用原生 `:title` 属性避免复杂性

3. **Composable 职责边界**
   - `useModal`: 仅状态管理（isOpen/open/close/toggle）
   - Focus trap 必须在使用 `useModal` 的组件层实现，不能放在 composable 里
   - 架构决策需要在 AC 中明确说明"谁做什么"

4. **Vue 3 Composition API 生命周期正确使用**
   - 全局事件监听（ESC keydown）必须在 `onMounted` 注册，`onUnmounted` 注销
   - 使用 `onActivated`/`onDeactivated` 处理 keep-alive 场景

### Process Insights

1. **验证必须实际执行**
   - `npm install` 必须先于 `npm run test`
   - 容器环境与本地环境不同，依赖可能不存在
   - 验证输出必须实际截图/记录，不能假设通过

2. **Code Review 质量保障有效**
   - UX-3 和 UX-5 的 Review 都发现了真实问题
   - Review 应该由不同 LLM 执行（新上下文）

3. **Epic Story 规划依赖管理得当**
   - UX-1 是基础，P0 优先级
   - UX-2/UX-3 可并行
   - UX-4 依赖 UX-1/2/3
   - UX-5 依赖 UX-1/2/3/4
   - 依赖链清晰，执行顺畅
