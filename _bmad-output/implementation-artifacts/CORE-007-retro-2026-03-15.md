## 概要

CORE-007 成功实现了应用的离线模式支持，包含网络状态检测、离线任务队列管理和前端状态指示器。用户在无网络环境下仍可正常使用截图保存、手动速记、历史记录查看等核心功能，AI 分析和报告生成请求自动入队，网络恢复后自动重试。新增 780 行 Rust 代码（2 个新模块），13 个单元测试，所有 299 个 Rust 测试和 191 个前端测试通过。

## 做得好的地方

### 1. 关注点分离的模块设计

初始实现将所有功能放在单个 `network.rs` 模块中（329 行），经过重构后拆分为两个独立模块：

| 模块 | 职责 | 代码行数 |
|------|------|----------|
| `network_status.rs` | 网络状态检测、缓存、后台监控 | 179 行 |
| `offline_queue.rs` | SQLite 任务队列、入队/出队、重试、清理 | 601 行 |

这种拆分使得每个模块职责单一，便于独立测试和维护。

### 2. 利用现有架构模式

- **全局状态管理**：复用 `DB_CONNECTION` 全局 Mutex 模式访问 SQLite，未创建独立连接
- **Tauri 事件系统**：使用 `app.emit()` 发送 `network-status-changed` 和 `offline-queue-updated` 事件，前端通过 `listen()` 接收
- **AtomicBool 缓存**：与 `AUTO_CAPTURE_RUNNING` 相同的模式，用 `NETWORK_ONLINE` 缓存网络状态
- **错误处理**：遵循项目的 `Result<T, String>` + `.map_err()` 模式

### 3. 健壮的重试机制

- **指数退避**：`2^retry_count` 秒（1, 2, 4, 8, 16s），最大 60 秒
- **最大重试次数**：5 次，超出后标记为 permanently failed
- **并发保护**：`QUEUE_PROCESSING` Mutex 防止多个处理器同时运行
- **网络中断检测**：处理队列时每个任务前检查网络状态，离线则暂停

### 4. 离线兼容性

- **截图功能**：离线时截图照常保存到本地，仅 AI 分析入队
- **手动速记**：离线时正常保存到本地 SQLite
- **日报生成**：离线时返回友好提示，引导用户联网后重试
- **历史查询**：SQLite 本地操作，天然支持离线

### 5. 测试覆盖全面

13 个新测试覆盖了核心场景：
- 网络状态缓存的读写和切换检测（6 个）
- 任务入队、完成、失败、重试耗尽、计数、清理（7 个）
- 使用 `#[serial]` 防止全局状态竞争

### 6. 前端集成简洁

在 `App.vue` 中新增约 56 行代码，通过事件监听 + 60 秒轮询 fallback 实现离线状态展示和待同步任务数量显示，未引入新组件，保持了前端架构简洁。

## 遇到的挑战

### 1. 初始实现需要重构

**过程**：
- 第一次实现（commit `10e29fd`）将网络检测和离线队列全部放在 `network.rs` 中（329 行）
- 发现职责过于集中后，重新实现为 `network_status.rs` + `offline_queue.rs`（commit `14a9eee`）
- 最后删除旧的 `network.rs`（commit `5689e64`）

**教训**：对于涉及多个关注点的功能（检测 vs 队列管理），应在设计阶段就规划模块拆分，而非先合并再拆分。

### 2. CI 环境中 xcap 编译问题

测试命令需使用 `--no-default-features` 避免 xcap 在无桌面环境下的编译失败。这是 CORE-002 时期遗留的已知问题，CORE-007 正确地延续了该解决方案。

### 3. 网络检测策略选择

**决策**：使用 reqwest HEAD 请求探测用户配置的 API 端点（或默认 `https://api.openai.com`），5 秒超时。

**权衡**：
- 直接探测 API 端点比 ping google.com 更有意义（API 可达才是真正的"在线"）
- 5 秒超时在弱网环境下可能产生误判
- 30 秒检测间隔在状态切换的响应速度和资源消耗之间取得平衡

### 4. 测试隔离

**问题**：多个测试模块共享全局 `DB_CONNECTION`，并行执行时可能产生竞争。

**解决**：为所有离线队列相关测试添加 `#[serial]` 属性，确保测试顺序执行。

## 经验教训

### 1. 离线功能的"降级而非禁用"策略

CORE-007 的核心设计原则是"离线降级"而非"离线禁用"：
- 截图照常保存，AI 分析入队
- 日报/周报/月报生成入队，返回友好提示
- 查询功能完全不受影响（SQLite 本地操作）

这种策略确保用户在任何网络条件下都能继续工作，仅 AI 依赖功能延迟处理。

### 2. SQLite 作为轻量级消息队列

使用 SQLite 表作为离线任务队列的优势：
- 无需引入外部依赖（Redis、RabbitMQ 等）
- 天然支持持久化，应用重启后队列不丢失
- 与现有数据库架构完美融合
- 支持索引和复杂查询（按状态、重试次数筛选）

### 3. 前端事件驱动 + 轮询 fallback 模式

使用 Tauri 事件实时通知 + 60 秒轮询作为 fallback，兼顾了实时性和可靠性。这是处理后端推送场景的良好模式。

### 4. 模块设计原则

新功能应创建独立模块，而非在现有模块中堆积。便于后续维护和测试。
