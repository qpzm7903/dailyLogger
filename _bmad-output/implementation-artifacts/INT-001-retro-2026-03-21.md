## 实现亮点

### 1. Markdown 转换器

使用 `pulldown-cmark` 库解析 Markdown，转换为 Notion Block 结构：

- 支持标题（H1-H6）→ `heading_1` ~ `heading_6`
- 支持有序/无序列表 → `numbered_list_item` / `bulleted_list_item`
- 支持代码块 → `code`
- 支持引用块 → `quote`
- 支持行内格式（粗体、斜体、行内代码）

### 2. RichTextBuilder 模式

实现了累积器模式处理行内格式，支持状态追踪：
- `set_bold(true/false)` - 粗体状态
- `set_italic(true/false)` - 斜体状态
- `set_code(true/false)` - 行内代码状态

### 3. 分块上传

实现了 `chunk_blocks` 函数，确保每次 API 请求不超过 100 blocks（Notion API 限制）。

### 4. 属性名自动检测

通过 `get_title_property_name` 查询数据库 schema 自动识别标题属性名，支持：
- Name, Title, 标题, 名称

## 开发过程中的挑战

### pulldown-cmark API 差异

`pulldown-cmark 0.9` API 与最新版本有差异，需要适配：

- `Tag::Heading` 是 3 字段元组变体，不是结构体
- `Tag::List(Some(n))` 表示有序列表，`None` 表示无序列表
- `Event::End` 接受 `Tag` 而不是 `TagEnd`

## 经验教训

### 1. 第三方库 API 验证

第三方库的 API 可能与文档不同步，需要实际编写测试验证。建议在开始实现前先编写简单的 POC 测试。

### 2. API 限制处理

外部 API 通常有请求限制（如 Notion 的 100 blocks），提前了解并实现分块逻辑很重要。

### 3. 累积器模式

对于需要状态追踪的转换逻辑（如行内格式），累积器模式是一个清晰的解决方案。
