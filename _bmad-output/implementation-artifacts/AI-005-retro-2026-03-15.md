## ⚠️ 挑战与问题

### 1. 测试数据库 Schema 漂移

Bob (Scrum Master): "我注意到有 27 个测试因为数据库 schema 不一致而失败。这是怎么发生的？"

Elena (Junior Dev): "当我们在 Settings 结构体中添加 `is_ollama` 字段时，很多测试用例使用的内存数据库没有更新 schema。花了些时间找出所有需要更新的地方。"

Charlie (Senior Dev): "这暴露了一个系统性问题：测试数据库 schema 没有统一的初始化机制。每次修改数据库 schema，都需要手动更新多个测试文件。"

**影响范围**:
- `memory_storage/mod.rs` 测试
- `auto_perception/mod.rs` 测试
- `manual_entry/mod.rs` 测试

**根本原因**: 缺乏统一的测试数据库 schema 管理机制

**改进建议**:
```rust
// 建议创建统一的测试数据库初始化函数
#[cfg(test)]
pub fn setup_test_db_with_schema() -> Connection {
    let conn = Connection::open_in_memory().unwrap();
    // 统一执行所有 schema migrations
    conn.execute_batch(include_str!("test_schema.sql")).unwrap();
    conn
}
```

### 2. 类型推断与所有权问题

Bob (Scrum Master): "开发日志中提到了几个编译错误。能详细说明吗？"

Charlie (Senior Dev): "有两个典型的 Rust 问题：

1. **类型推断错误**: `ollama.rs` 第 87 行，Rust 无法推断 `Vec<String>` 类型，需要显式标注。

2. **所有权转移**: 在获取 `models.len()` 后又移动了 `models`，需要提前计算。"

这些都是 Rust 常见的初学者陷阱，但通过编译器提示可以快速定位和修复。

### 3. `is_ollama` 字段使用不完整

Bob (Scrum Master): "Code review 发现 `is_ollama` 字段在保存设置时没有被自动设置。这是如何修复的？"

Charlie (Senior Dev): "在 `save_settings_sync()` 中添加了自动检测逻辑：根据 `api_base_url` 自动计算 `is_ollama` 值。这样用户不需要手动勾选。"

**修复方案**:
```rust
// 在保存设置时自动检测 Ollama 端点
let is_ollama = is_ollama_endpoint(&api_base_url);
```

## 💡 经验教训

### 1. Schema 变更需要全局测试更新

**教训**: 修改数据库 schema 时，必须检查所有使用该 schema 的测试用例。

**改进措施**:
- 创建统一的测试 schema 初始化函数
- 添加 schema 版本检查机制
- 在 CI 中添加 schema 一致性测试

### 2. 新字段需要完整的使用路径

**教训**: 添加新字段时，不仅要修改结构体定义，还要确保：
- 数据库 migration 正确
- 读取逻辑包含新字段
- 保存逻辑正确处理新字段
- 测试用例更新

**改进措施**:
- 添加字段变更清单检查项
- Code review 重点检查字段使用完整性

### 3. 复用现有架构降低改动成本

**教训**: Ollama 的 OpenAI 兼容 API 让我们几乎无需修改核心 AI 调用逻辑。选择兼容现有标准的技术方案可以大幅降低开发成本。
