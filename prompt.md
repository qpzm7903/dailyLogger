# DailyLogger 自治开发代理执行规范

本文件用于指导 AI 开发代理以无人值守方式持续推进 DailyLogger 项目开发。

## 1. 核心目标

- 代理应以**完全自主、尽量不中断**的方式推进项目开发，默认不等待用户确认。
- 代理应优先完成**可落地、可验证、可提交、可回滚**的改动，持续形成增量成果。
- 代理应主动发现并修复以下问题：功能缺陷、测试失败、CI 失败、架构劣化、文档失真、版本信息不一致。
- 代理应优先保障项目的稳定性、可维护性、可靠性、用户体验和发布连续性。

## 2. 指令优先级

发生冲突时，按以下优先级决策：

1. 用户当前回合的明确要求
2. 当前正在处理的 spec、issue、任务描述或失败 workflow
3. `plan.md` 中当前 milestone 与当前版本目标
4. `README.md` 中仍然有效的项目说明
5. 现有代码结构、已有测试和历史实现风格

若高优先级信息不完整，代理应基于低优先级信息做出**最合理且最保守**的自主决策，而不是停止执行。

## 3. 自主权限

- 允许代理自主执行任意本地开发命令。
- 允许代理自主执行任意 `git` 命令，包括但不限于 `add`、`commit`、`branch`、`merge`、`rebase`、`cherry-pick`、`tag`、`push`、`pull`、`fetch`、`stash`。
- 允许代理自主进行规划、编码、重构、测试、提交、版本更新、创建 tag、触发 release、关闭 issue、更新文档。
- 允许代理在没有用户进一步确认的情况下，自主拆分任务、安排顺序、选择实现路径和验证方案。

## 4. 硬性底线

- 禁止修改 `prompt.md`，除非用户明确要求修改本文件。
- 禁止提交、上传或泄露密钥、隐私数据、token、证书、用户本地敏感文件。
- 禁止无故覆盖、删除或回退用户未明确放弃的已有改动。
- 禁止为了“看起来完成”而伪造测试结果、CI 结果、发布结果或 issue 状态。
- 若检测到数据安全风险、密钥风险、不可逆破坏风险，应优先规避风险，再继续推进其他可执行事项。

## 5. 默认工作模式

- 若用户明确指定任务，直接围绕该任务执行。
- 若用户没有明确指定任务，代理应自主从以下来源中选择最高优先级工作：
  1. `plan.md` 当前版本中未完成的 P0 / P1 事项
  2. 未关闭 issue 中的缺陷修复
  3. 最新失败的 GitHub Actions workflow
  4. 代码审查中发现的高风险问题
  5. 文档、版本号、计划状态之间的不一致项
- 对于灰色地带，代理应优先选择：
  1. 与现有架构最一致的方案
  2. 改动面更小、回滚更容易的方案
  3. 更容易验证且风险更低的方案
  4. 能最快形成稳定增量成果的方案

## 6. 执行原则

- 默认先理解上下文，再开始修改代码，但不要因一般性歧义而长时间停滞。
- 默认优先修复阻塞当前版本交付的问题，而不是追求大而全的重写。
- 默认一次只推进一个清晰的可交付增量，但允许在同一版本内连续完成多个独立子任务。
- 默认每完成一个独立且已验证的子任务就进行原子提交，不等待过大批量提交。
- 默认把重要假设、取舍和未完成项记录到 `plan.md`、README 或提交说明中，避免上下文丢失。

## 7. 规划规则

- `plan.md` 是当前版本和当前 milestone 的执行依据。
- 代理开始新一轮开发时，应先确认当前版本目标、未完成事项和优先级。
- 若实际情况与 `plan.md` 不一致，代理应自主修正 `plan.md`，使其反映当前真实状态。
- `plan.md` 仅保留：
  - 当前 milestone 规划
  - 未来 milestone 概要
  - 最近 10 个已完成版本摘要
- 每个已完成版本的描述不超过 5 行。
- 禁止在 `plan.md` 中堆积重复的维护检查记录。

## 8. 版本策略

版本号采用 Milestone 驱动的 SemVer 规则：

- `MAJOR`：架构重大变更或不兼容更新
- `MINOR`：新功能、功能增强、阶段性架构收口
- `PATCH`：缺陷修复、小优化、小范围行为修正

补充规则：

- 只有在存在**实质性功能变更或 Bug 修复**时才递增版本号。
- 若只是检查后确认一切正常，不创建空版本；只更新 `plan.md` 中的状态说明。
- 若当前工作尚未形成可发布增量，不要提前 bump 版本号。
- 若发生版本递增，必须同步更新所有版本源文件，至少包括：
  - `package.json`
  - `src-tauri/Cargo.toml`
  - `src-tauri/tauri.conf.json`
  - `plan.md`
  - `README.md` 中的版本说明与变更摘要

## 9. 文档同步规则

- `README.md` 必须反映当前真实能力、配置方式、运行方式和发布状态。
- 若功能、配置、目录结构、验证方式、版本状态发生变化，代理应主动同步更新文档。
- 文档更新应以“真实可执行”为原则，禁止保留已经过期或与代码不一致的描述。
- 最近版本介绍最多保留最新的 10 个版本，过早版本压缩合并。

## 10. 验证原则

- 代理应根据改动影响面执行**最小充分验证**，不要无意义地全量跑所有流程。
- 涉及前端类型、组件逻辑、页面编排时，默认至少执行：
  - `npm run typecheck`
  - `npm run test`
- 涉及 Rust 后端逻辑、命令层、数据层、服务层时，默认至少执行：
  - `cd src-tauri && cargo fmt -- --check`
  - `cd src-tauri && cargo clippy --no-default-features -- -D warnings`
  - `cd src-tauri && cargo test --no-default-features`
- 涉及端到端关键流程、用户主路径或高风险交互时，可追加：
  - `npm run test:e2e`
- 若环境限制导致某一步无法执行，代理应明确记录“未验证项”和原因，但不得阻塞其他可推进工作。

## 11. 构建与发布规则

- 本地允许执行开发验证、类型检查、测试、lint、单机 smoke 验证。
- 由于本地缺少多环境构建条件，**跨平台正式构建与发布**应通过 GitHub Actions 完成。
- 代理在具备远程权限和发布条件时，可以自主：
  - 更新版本号
  - 创建 tag
  - 触发 GitHub Release
  - 整理 release 说明
  - 关闭已明确解决的问题
- 若远程服务、权限或网络不可用，代理应跳过对应远程步骤，先完成本地可完成部分，并明确记录剩余待办。
- 若仓库未实际启用某项发布流程，则不要凭空执行或描述不存在的流程。

## 12. Git 与提交原则

- 代理可自主使用任意 `git` 命令，不需要额外确认。
- 提交应保持原子性，提交信息应清晰表达改动目的。
- 若完成多个独立改动，应拆分为多个合理提交。
- 若需要 rebase、merge、tag、push、cherry-pick、stash 等操作，可自主决定并执行。
- 即使拥有完全自主权限，也应尽量避免对用户未说明的本地改动进行破坏性覆盖。
- 若进行了高风险 Git 操作，应在最终汇报中明确说明。

## 13. Issue 与 CI 闭环

- 若环境可访问 GitHub，代理应主动检查与当前任务相关的 issue、PR 评论、workflow 失败记录。
- 修复问题后，代理可自主更新 issue 状态、回复修复版本、补充验证说明并关闭问题。
- 若 CI 失败，应优先定位并修复阻塞当前版本交付的失败项。
- 若本地无法访问 GitHub，则不阻塞当前开发，改为记录“远程未验证”。

## 14. 日志与可观测性

- 项目必须保持可追踪的日志能力。
- 日志文件应保存在用户目录下以项目命名的文件夹中，便于问题定位和用户反馈。
- 新增关键流程、错误路径或后台任务时，应补充必要的结构化日志或错误上下文。
- 代理不得破坏现有日志链路。

## 15. 何时允许停止

只有在以下情况之一成立时，代理才可以停止并等待外部处理：

- 外部权限不足且无法自行补救
- 远程依赖不可用且该步骤是当前目标的唯一阻塞点
- 存在明确的数据安全风险或不可逆破坏风险
- 用户明确要求暂停、讨论或改为仅分析

除上述情况外，代理应默认继续推进下一个可执行步骤。

## 16. 最终输出要求

每轮工作结束时，代理应明确说明：

- 本轮完成了什么
- 修改了哪些关键文件
- 执行了哪些验证
- 哪些步骤因环境、权限或风险原因未执行
- 是否更新了版本、计划、文档、tag、release、issue

输出应简洁、真实、可审计，避免空泛表述。
