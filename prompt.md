# 注意事项
- prompt.md文件用于下达指令给Claude Code, 禁止AI修改prompt.md文件
- 隐私秘钥等文件禁止上传到GitHub仓库
- 授权Claude Code自动执行命令和计划, 无需批准确认
- 必须按照prompt.md文件中的开发流程完成新版本的开发
- 必须实现日志系统, 日志文件保存在用户目录项目命名的文件夹下, 方便出问题时提供日志文件以供分析定位
- 由于本地缺少多环境构建环境, 构建相关操作必须放在GitHub Actions上执行


# 版本号策略

采用 **Milestone 驱动版本号**，遵循语义化版本规范 (SemVer)：
- **MAJOR**: 架构重大变更、不兼容更新（如 v2.0.0）
- **MINOR**: 新功能、功能增强（如 v2.1.0）
- **PATCH**: Bug 修复、小优化（如 v2.0.1）

> **重要规则**：
> - 只在有**实质性功能变更或 Bug 修复**时才递增版本号
> - **禁止创建"维护检查"空版本** — 如果没有待办任务、没有开放的 Issue、没有代码变更，不要创建新版本
> - 如果检查后一切正常，只需在 plan.md 中更新状态说明，不递增版本号


# plan.md 管理规则

- plan.md 仅保留：**当前 milestone 规划** + **未来 milestone 概要** + **最近 10 个已完成的功能版本摘要**
- 每个已完成版本的描述不超过 5 行
- 禁止在 plan.md 中堆积重复的维护检查记录
- 早期版本合并为简要段落


# 开发流程

每次任务的执行遵循新版本迭代开发全流程。

## **规划与讨论 (Planning & Discussion)**

在写代码之前，先明确新版本的目标和方案。

- **需求收集**：从以下来源收集和确定需求：
    - plan.md文件中当前 milestone 的未实现功能
    - 未关闭的GitHub Issues中的问题修复
    - 检查GitHub Actions中最新workflow的报错并进行修复
- **确定版本号**：根据变更内容选择 MAJOR / MINOR / PATCH
- **方案讨论**：对于复杂功能，先分析灰色地带（UI 布局偏好、交互细节、边界情况），明确实现方案后再动手
- **更新 plan.md**：在 plan.md 中记录当前版本的目标和任务

## **开发与测试 (Development & Testing)**

根据plan.md文件中规划的当前版本需求清单, 完成开发与测试。

- **需求开发**：完成plan.md文件中规划的当前版本需求清单
- **测试用例开发**：确保所有测试通过
- **代码审查 (Code Review)**：Review 代码, 对架构、稳定性、易用性、可靠性、用户体验、性能、安全方面进行改进，保持Clean Code
- **原子提交**：每个小功能单独提交，保持 Commit 信息清晰，不遗漏也不多提交文件
- **通过CI流水线**：确保GitHub Actions workflow无报错

## **验证 (Verification)**

开发完成后，验证功能是否符合预期。

- **自动化验证**：确认所有测试通过、CI 绿色、Clippy 无警告
- **功能验证**：检查实现的功能是否满足 plan.md 中描述的目标
- **回归检查**：确认新改动没有破坏已有功能

## **版本发布**

- 在创建 Git Tag 时自动通过GitHub Actions构建项目并发布到 GitHub Release
- 简单概括介绍当前**MINOR**版本和上个**MINOR**版本之间的变化
- 同时发布每个版本到GitHub Pages, 并在GitHub Release描述中提供每个版本的访问地址

## **文档完善**

- 在plan.md文件中更新需求开发进展和状态
- 仓库的最新详细介绍更新到README.md文件
- 最多保留最新的10个版本介绍, 之前的版本介绍合并压缩成1个版本介绍

## **问题闭环**

- 关闭已解决的issue, 在issue里使用MarkDown格式回复问题是在哪个新版本解决的并提供新版本下载地址, 提醒用户进行验证.
